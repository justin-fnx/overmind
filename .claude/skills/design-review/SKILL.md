# Figma 디자인 리뷰 (BOMAPP · 시니어 UX/UI 기준)

디자인 시안이 (1) **지시서를 충실히** 구현했고 (2) **빌드 규칙**을 지켰으며 (3) 사용자가 **읽고 쓰기에 불편이 없는지**를 근거 있게, 엄격하게 판정한다. **"통과"를 기본값으로 두지 않는다** — 실제 문제를 적극적으로 찾는다.

> 🧪 **객관성 원칙**: 가능하면 **빌더가 아닌 별도 Opus 서브에이전트**로 리뷰한다(자기 작업을 관대하게 채점하지 않도록). 빌더 컨텍스트를 상속하지 말고, 아래 입력을 처음부터 직접 수집해 판단한다. (`Agent` subagent_type=general-purpose, model=opus.)

---

## STEP 0 — 입력 수집 (직접 확인, 추측 금지)

1. **작업지시서**: `notion-fetch`로 태스크 본문(+본문 멘션/하위 페이지)을 읽어 요구사항을 구조화 — 화면 수·섹션·콘텐츠·상태(정상/예외/완료)·문구·**「결정 필요/미정」 항목**.
2. **빌드 규칙**: 대상이 따랐어야 할 스킬을 읽는다 — `.claude/skills/design-to-figma/SKILL.md` + 메모리 `reference_bomapp_bds_figma_design`.
3. **시각 입력(1차)**: 각 화면 노드를 **고해상도** `get_screenshot`(maxDimension 1500+) → URL `curl` → `Read`로 **전 화면을 눈으로** 본다. (빌더가 미리 렌더한 PNG가 있으면 그것도 활용.)
4. **실측(2차)**: 의심 지점은 `get_design_context`/`get_metadata`로 **폰트 크기·색 hex·여백·노드 구조**(네이티브/컴포넌트 여부)를 실측해 근거로 삼는다. (시각 판단이 1차, 실측이 근거.)
   - Figma/Notion MCP는 ToolSearch로 로드: `select:mcp__claude_ai_Figma__get_screenshot,mcp__claude_ai_Figma__get_metadata,mcp__claude_ai_Figma__get_design_context,mcp__notion__notion-fetch`.
   - ⚠️ **폰트 함정**: Figma MCP 환경엔 Pretendard가 없어 미리보기가 Noto Sans KR로 보인다 → **"폰트 렌더 자체"는 결함으로 깎지 말되, 실제 크기·대비·가독성은 엄격히** 본다.

---

## STEP 1 — 차원 1: 작업지시서 충실도

- **요구항목 ↔ 구현 대조표**를 만든다. 누락·오해·임의 변경을 찾는다.
- ⚠️ 지시서가 **"기획 확인 필요/미정"으로 남긴 항목**을 디자이너가 단독 확정하지 않았는지 — 했다면 "이 안으로 진행"임을 **에스컬레이션 목록**에 올린다(그래프 형태·레이블·완료 방식·최종 문안 등).
- 명문 **카피를 임의 변경**했는지, **변수화(템플릿) 대상**(예: 상태값 정상/주의/위험)이 헤드라인에 하드코딩돼 변수화가 막히지 않는지.

## STEP 2 — 차원 2: 빌드 스킬 준수도

대상 SKILL의 절대 규칙을 **규칙 ↔ 준수여부 대조표**로. BOMAPP `design-to-figma` 기준 최소 점검:

- 화면 전체가 **네이티브 노드**(평면 이미지 업로드 아님) · **반복요소 = 실제 Figma Component + 인스턴스**(동일 역할인데 일회성 프레임으로 샌 곳 없는지) · **배경(기기) 프레임 + cornerRadius + 그림자** · **최소 높이 375×812 + 하단요소(CTA·trust·약관) 도킹** · **BDS 토큰** 사용 · **Pretendard 스왑 명시** · **버전 페이지 분리(기존 보존)** · **차트 = 상태 컬러밴드** · 컴포넌트 프로퍼티(label/showArrow/title 등).

## STEP 3 — 차원 3: UX/UI 시니어 휴리스틱

각 카테고리를 평가하고 발견을 심각도 태그로 남긴다.

**🎨 색 대비 (WCAG 2.1 AA — 가장 자주 터지는 결함, 반드시 수치로)**
- 기준: 본문 **4.5:1**, 큰글자(≥24px, 또는 ≥18.66px bold) **3:1**, UI/그래픽 경계 **3:1**.
- 의심 쌍은 **hex로 상대휘도 대비비를 직접 계산**해 수치 제시(배경을 PNG에서 추정했으면 "추정" 표기).
- 🔴 **BOMAPP 상습 결함**: 채도 높은 상태색 **warn `#FF8A00`·safe `#12B886`을 텍스트·수치로** 쓰면 밝은 배경(`#F3F5F8`/약배경)에서 **~2.1~2.4:1** → AA·큰글자 기준 모두 미달(핵심 메트릭·상태 Pill·강조 문구가 여기 걸림). → **텍스트 전용 darker 변형**(오렌지→`#C2410C`대, 그린→`#0E7C5A`대, ≈4.5:1+)을 따로 두고, **채도색은 막대·아이콘 등 비텍스트 그래픽에만** 한정. 회색 `#8A94A6` 캡션도 흰/연배경서 ~2.8~3.1:1 미달 → 본문급은 `#3B4252`(ink2)로. primary `#3C7AE5`도 작은 텍스트(eyebrow·링크)면 ~3.8:1 → 텍스트용 한 톤 darken.
**♿ 접근성**: 상태를 **색만으로** 전달하지 않는지(색 + 텍스트 라벨 이중 인코딩이 기본기). 명도 대비는 색각이상과 **별개**로 따로 본다.
**🪜 정보 위계 · 가독성**: 스캔 가능성, "한 화면 한 메시지", 시선 흐름.
**🔤 타이포그래피**: 본문 **≥14**, 캡션 **≥11~12**(9px 같은 초소형 금지), 라인하이트 1.4~1.6, 줄길이.
**📐 여백 · 밀도**: 너무 좁아 답답하거나, 너무 비어 허전한 구간(특히 **콘텐츠 ~ 하단 CTA 사이 빈 공간**) — 짧은 화면은 콘텐츠 세로 중앙정렬 권장.
- **🃏 카드/컨테이너 *내부* 패딩(놓치기 쉬움 — 반드시 실측)**: `get_metadata`로 카드 자식들의 x/y/크기를 떠서 (a) **카드 간 내부 패딩 일관성**(예 16~20px 통일, 카드마다 14/18/32로 제각각이면 흠), (b) **콘텐츠 오버플로우** — 절대배치 자식의 `y+height`가 카드 `height`를 넘으면 박스가 카드 밖으로 삐져나옴(plain frame은 clip 안 됨 → 흔한 버그), (c) 자식 박스가 카드 모서리에 **딱 붙어 패딩이 0/음수**가 아닌지를 점검한다. (시각만으론 놓치기 쉬워 좌표 실측이 필수.)
**🧩 일관성**: 동일 역할 = 동일 컴포넌트/토큰(라운드·간격·색).
**👆 어포던스 · 터치타겟**: 버튼/링크 인지성, **≥44×44pt**(텍스트 링크 히트박스 포함). 읽기전용 필드가 "입력 불가"로 읽히는지.
**📊 데이터 시각화**: 직관성·오해 없음·축/범례/단위. ⚠️ **비교 막대는 값 비례**여야 한다(0원과 큰 값의 막대 폭이 같으면 "길이=금액" 오독).
**✍️ 콘텐츠 · 마이크로카피**: 명확성·톤·**용어 일관**(예: "건강 데이터" vs "보험 연결" 혼용 금지).

## STEP 4 — 보고서 (한국어, 이 형식 그대로)

1. **총평 & 판정**: `만족` / `조건부 만족` / `미흡` + 2~3문장 요약 + **화면별 한 줄 인상**.
2. **차원 1 / 2 / 3** 각각: **대조표**(해당 시) + 발견 목록. 발견마다 → `[Critical]/[High]/[Medium]/[Low]/[Nit]` 심각도 + **위치(화면·요소)** + 문제 + **구체적 권장 수정** + **근거(대비비·px 등)**.
3. **우선순위 수정 Top 5~8** (심각도순, 화면·요소·한 줄 조치).
4. **잘된 점 3~5** (균형 — 무엇을 지킬지).
5. **기획 에스컬레이션**: 시안 확정 전 컨펌 필요 항목 목록.

### 심각도 기준
- **Critical**: 사용 불가 / 지시서 핵심 누락 / 심각한 접근성 차단.
- **High**: 핵심 정보가 AA 대비 미달 / 주요 요구 미흡.
- **Medium**: 가독성·일관성 저하 / 보조정보 대비 미달 / 미확정 항목 단독 결정.
- **Low·Nit**: 개선 권장 / 미세.
