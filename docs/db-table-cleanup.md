## 2. 테이블 판정 결과 (BOM-207)

> **핵심 교훈: "정적 코드 참조 미존재 ≠ 드롭 가능"**.
> 아래 세 가지 함정을 반드시 교차 검증한다.
> 1. **법적 보관의무** — 코드가 안 써도 법령상 보관해야 하는 호출이력 (마이데이터 등).
> 2. **런타임 dead code 잔재** — 정적 참조는 있으나(KEEP처럼 보이나) 그 참조 코드 자체가 죽은 코드라 실제로는 드롭 대상.
> 3. **사용자 노출 잔존** — 과거 발송 콘텐츠(알림톡 게이트페이지 등)를 사용자가 다시 열어볼 수 있어 유지해야 하는 데이터.

## 3. 마이데이터 호출이력 보관 정책 (BOM-217)

### 3.1 법령 근거

- **근거**: 신용정보법 시행령 **제18조의6 제10항** — 본인신용정보관리업자의 개인신용정보 제공·이용 기록 **작성·보관** 의무.
- **법령상 명시 보관기간**: 없음.
- **내부 보관기준**: **5년 (보수적)** — 민원·분쟁·감독대응 및 전송내역 통지 근거 유지 목적 (신정민 의견). ⚠️ 내부 추정치이므로 **보안팀/마데 운영 컨펌 필요**.
- **감독 대응**: 금감원이 **월·분기·연도별**로 호출이력 자료를 요청. 보고 시점에 즉시 쿼리 가능한 상태(hot) 유지 필요.

### 3.2 v1 / v2 구분 — 가장 중요한 함정

| 테이블 | 상태 | 처리 |
|---|---|---|
| `log_my_data_api_request` (v1, suffix 없음) | 응답속도 이슈로 v2 신설 후 **적재 중단된 구버전**. 금감원 보고 쿼리(노션 DW013/DW014)에서 **직접 참조 안 됨**. | 코드 미참조여도 **법적 보관의무 대상 → 드롭 불가**. 보관 정책만 적용. |
| `log_my_data_api_request_v2` | 현재 적재·운영·보고에 활발히 사용. 노션 **DW013/DW014 금감원 보고 쿼리가 전부 v2 참조**. | **파티셔닝 + 주기적 정리의 실제 타겟.** |

> 재은 님이 티켓에서 지정한 테이블명은 v1이지만, 실제 적재가 지속되어 파티셔닝이 의미 있는 건 v2다. BOM-217은 이 구분을 명시하고 실타겟을 v2로 잡았다.

### 3.3 파티셔닝 / 정리 전략 (BOM-217 권고안)

- **파티션 키**: `created_at` 기준 **RANGE 파티셔닝**.
- **파티션 단위**: **월별(monthly)** — 금감원 보고가 월/분기/연 단위라 월 파티션이면 분기·연 집계 시 파티션 프루닝이 정확히 맞아떨어짐. v2는 데이터량이 커 연 단위 파티션은 과대.
- **Hot 보관**: 최근 **13개월**(보고 시점 여유 1개월 포함) 메인 파티션 유지.
- **정리 방식**: 5년 경과 파티션은 `DROP PARTITION`으로 즉시 삭제(개별 `DELETE` 대비 락·부하 최소). 13개월~5년 구간은 유지하되, 콜드 스토리지(S3/아카이브 테이블) 이관은 후속 백로그로 분리.
- **자동화**: 매월 1일 신규 파티션 생성 + 만료 파티션 DROP을 이벤트 스케줄러 또는 배치(n8n/cron)로 운영.
