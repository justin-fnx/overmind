## 0. 열린 질문 권고안 (상위 §14 — 승인 대기)

| # | 질문 | 권고 | 근거 |
|---|------|------|------|
| Q1 | "검진" 명칭 | **워킹타이틀 "보험 검진" 유지, 대외 명칭은 법무 확인 후 확정** (대안 1순위: "보장 점검") | 의료 오인 리스크는 명칭만의 문제라 마지막에 갈아끼울 수 있음. 프레임(문진→결과지→처방)은 명칭과 무관하게 유효 |
| Q2 | L0 코호트 데이터 | **공개 통계 조합으로 시작** — 가계금융복지조사(통계청 MDIS) + 보험개발원·보험연구원 공표 통계. 자체 익명 통계는 Phase 3에서 법무 검토 후 | 출시 속도 + 법무 리스크 제거. L0 용도로는 거친 셀(연령×성별×소득분위)로 충분 — 정확도 게이지가 한계를 정직하게 표시해 주는 구조라 거칠어도 성립 |
| Q3 | 웹→앱 동선 | **간이 검진 = 설치 없는 독립 웹, 정밀 검진(L3) = 본앱 합류** 확정 권고 | L0의 존재 이유가 문턱 제로. 앱 설치는 연동보다 앞선 허들 |
| Q4 | 자기 신고↔실계약 교정 수위 | **"축하 프레임"**: "실제 계약을 반영해 3곳이 더 정확해졌어요" — 다름의 지적이 아니라 정확도 상승으로 서술 | 무안 주기 방지. 교정 상세는 접힌 영역("무엇이 달라졌나요")에 |

| Q6 | Wings B2B2C | **보류.** Phase 2 안착 후 재논의 | 소비자向 신뢰 자산이 먼저 쌓여야 설계사向 제공도 설득력 있음 |

## 1. MVP 스코프

### In
- **정밀도 사다리 L0~L2** (지연 문진, 총 문항 ≤ 10)
- **위험 점검 영역: 암 1개** — 내러티브 시나리오는 성별 기반 2종 (남: 위암 / 여: 유방암), 판정은 암 영역 통합
- **결과지**: 암 영역 판정(월급 공백/치료비 2행) + 미검진 4행(뇌·심혈관/사망/장해/장기요양) + 정확도 게이지 + 처방 2줄 (✂️정리 / 🛡️보강)
- 💰(놓친 보험금) 슬롯은 **잠금 표시 + "정밀 검진 오픈 알림 신청"** — Phase 2 수요를 사전 등록으로 측정
- "버틸 수 있는 금액" 슬라이더 + 실시간 재판정
- 결과지 저장 (URL 재방문 토큰) — 회원가입 없이

### Out (명시적 제외)
- 마이데이터 연동 일체 (L3), 놓친 보험금 스캔 (M6)
- 뇌심혈관·사망·장해 영역, M4 진짜 가격 비교기, M5 실질가치 타임라인
- 회원 체계 (기존 BOMAPP 계정 연결 포함), 상품 연결 (처방 🛡️는 "이런 유형이 필요하다"까지만 — 특정 상품 비노출로 모집 규제 회피)
- 몬테카를로, 푸시/알림톡

## 3. 계산 스펙 v1 (암 영역)

### 3.1 입력 변수
| 변수 | 출처 | 폴백 (미입력 시) |
|------|------|----------------|
| age, sex, income_band | L0 | — (필수 3문항) |
| liquid_assets | L1 | 코호트 p50 |
| household (외벌이/맞벌이, 부양가족 수) | L1 | 코호트 최빈값 |
| has_silson (실손 유무), dx_benefit (암 진단비 금액), premium_monthly | L2 | 코호트 가입률 기반 기대값 ("모름" 동일) |
| capacity_adj (슬라이더 조정값) | S5 | 시스템 제안값 |

### 3.2 수식
```
# 버틸 수 있는 금액 (시스템 제안값)
capacity = liquid_assets × (1 − safety_margin)                    # safety_margin 기본 0.2
         + spouse_income_months × treat_months × (맞벌이 ? 1 : 0)  # 위기 중 유지 소득 여력

# 총손실 (k ∈ {p50, p90})
med_self(k)   = med_covered_self(k) + nonpay(k)                   # 산정특례 반영 급여 본인부담 + 비급여
med_net(k)    = max(0, med_self(k) − silson_coverage(k))          # 실손 반영 (has_silson 시)
income_loss(k)= treat_months(k) × monthly_income × income_stop_ratio
loss(k)       = med_net(k) + income_loss(k) + care_cost(k)

# 판정
gap = loss(p90) − capacity_adj − dx_benefit_effective
    where dx_benefit_effective = dx_benefit × (1 − claim_miss_penalty(금액구간))
gap > 0            → 🔴 부족 (빈 곳 = gap)
gap ≤ 0 이고
  소액보장(한도 내 손실 대응 특약) 존재 → 🟡 넘침 후보 (자기 신고 기반이라 "검토" 수위로만)
  그 외                              → 🟢 적정
```

### 3.3 파라미터 테이블 (정적 데이터, 버전 관리)

**cohort_profile** — M0 코호트 추정 엔진
| 컬럼 | 예 |
|------|-----|
| age_band(5세)×sex×income_quintile | 35-39 · M · Q3 |
| liquid_assets_p50, household_dist | 2,800만, {외벌이 0.42, …} |
| silson_rate, dx_benefit_avg, premium_monthly_avg | 0.71, 1,900만, 21만 |
| source, version | 가계금융복지조사 2025, v2026.1 |

**disease_event** — 질병 이벤트 파라미터
| 컬럼 | 예 (위암) |
|------|----------|
| disease, stage_mix | stomach_cancer, {국한 0.6, 국소 0.25, 원격 0.15} |
| treat_months_p50/p90 | 8 / 14 |
| med_covered_self_p50/p90, nonpay_p50/p90 | 300/600만, 700/2,500만 |
| care_cost_p50/p90, income_stop_ratio | 300/1,200만, 0.9 |
| source, version | 암등록통계+건보공단 2025, v2026.1 |

**claim_miss_penalty** — 청구 누락 확률 (금액 구간별; 소액일수록 높음). 초기값은 문헌·내부 추정 → 추후 M6 실데이터로 교정.

- 모든 테이블에 `source`·`version` 컬럼 필수 (모델 문서 공개 원칙과 한 몸).
- 갱신 주기 연 1회 + 수시 (통계 개정 시). 관리 주체: 운영 어드민(bomapp-console에 파라미터 관리 화면 — Phase 1은 시드 데이터 하드로드로 충분, 콘솔 화면은 Phase 2).

### 3.4 검증 절차 (출시 전)
- 대표 페르소나 12종 (연령 4 × 소득 3) 수기 계산 대조 — 계리 자문 1회 검수 권장.
- 경계 테스트: 자산 0 / 고자산(캐패시티 > p90 손실 → 전 영역 🟢 + "보험 없이도 감당 가능" 소견이 실제로 출력되는지 — **D1의 리트머스 시험**).

| 4 | 셀 그리드 구성: 연령 5세(20~69) × 성별 × 소득 5분위 = 최대 100셀. 희소 셀은 상위 집계(10세·분위 병합) 폴백 | cohort_profile v1 |
| 5 | 소스·버전 명세 문서화 (모델 공개 문서의 부록) | 데이터 계보(lineage) 문서 |

리스크: 마이크로데이터 이용 승인 리드타임(수 주) → 1차는 공표 집계표만으로 거친 버전 구성 가능 (정확도 게이지가 한계를 흡수).

## 5. 기술 배치 초안 (BOMAPP 아키텍처 관점)

> 결정 아님 — 아키텍처 논의용 옵션 정리. 실행 시 Linear 에픽 + 서비스별 Task로 분해.

### FE — 독립 웹 (Q3 권고 반영)
- **next-frontend 모노레포에 신규 앱** (예: `checkup-web`) 권고. 기존 Vue3+Vite+Tailwind 컨벤션·공유 컴포넌트 재사용, 배포 파이프라인 동형.
- 비로그인·설치 없음·모바일 우선. 결과지 저장은 서명된 재방문 토큰 URL (개인정보 최소화).
- 공유·SEO가 중요해지면 SSR 검토 (planner-card-ssr의 EC2 capacity 교훈 참조 — 초기엔 정적 SPA로 충분).

### BE — 계산 API
| 옵션 | 내용 | 평가 |
|------|------|------|
| a. next-backend 기존 앱에 도메인 모듈 추가 | 비로그인 public 성격상 `open-api` 쪽이 인증 모델 부합. `bomapp-api`는 회원 전제라 부적합 | **Phase 1 권고.** 신규 ECS 앱 없이 시작 — 파라미터 테이블+계산 로직은 순수 도메인 모듈로 격리해 추후 분리 용이하게 |
| b. 신규 server 앱 (`checkup-api`) | 독립 배포·스케일 | Phase 2(연동·스캔 워크로드 합류) 시점에 재평가 |
| c. FE 클라이언트 계산 | 서버 불요 | 모델 파라미터·판정 로직이 통째로 노출, A/B·모델 갱신 불리 — 비권고 |

- 저장 데이터: 세션 응답(익명, 재방문 토큰 키) + S13 알림 신청 연락처(별도 동의)만. 회원 테이블 미사용.
- 파라미터 테이블: 코드 배포와 분리된 버전드 리소스 (S3 JSON 또는 DB 테이블 + 캐시).

### 계측
- Amplitude (next-frontend 기존 스택) — 화면별 퍼널 + 계단 상승 이벤트 + 슬라이더 조정 이벤트.
